### 問題点の整理

移行にあたって解決すべき課題は3つです。

**① サイトURLのハードコーディング**
現状は `gsc_mcp_server.py` に `SITE_URL` が直書きされています。ユーザーが管理する複数サイトから選択できる仕組みが必要です。

**② MCPのプロセス通信問題**
現状のMCPはstdioで動いており、サーバープロセスとして常駐させる必要があります。Vercelはサーバーレス（リクエストごとに起動・終了）なのでstdio MCPはそのまま使えません。GSC APIの呼び出しをNext.jsのAPI Routesに直接組み込む形に書き直します。

**③ 認証とGSCトークンの紐付け**
ClerkのユーザーIDに対してGSCのOAuthトークンを安全に保存・取得する仕組みが必要です。

### 実装工程

#### Phase 1 — Next.js プロジェクト基盤
**Step 1** : Next.jsプロジェクト作成・Clerk導入・基本的なログイン/ログアウトの実装

#### Phase 2 — GSC OAuth連携
**Step 2** : GSC OAuth フロー実装（ユーザーがGoogleアカウントを連携するUI＋API Route）
**Step 3** : OAuthトークンのDB保存（ClerkのユーザーIDをキーにSupabaseへ保存）

#### Supabase DB設計（Step 3）

`gsc_oauth_tokens` テーブルでユーザーごとの最新トークンを1行で管理します。

```sql
create table if not exists public.gsc_oauth_tokens (
  clerk_user_id text primary key,
  access_token text not null,
  refresh_token text,
  scope text not null,
  token_type text not null,
  expires_at timestamptz not null,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index if not exists idx_gsc_oauth_tokens_expires_at
  on public.gsc_oauth_tokens (expires_at);
```

運用方針:
- OAuth callback時に `clerk_user_id` をキーに `upsert` して最新化
- API呼び出し前に `expires_at` を確認し、期限切れなら `refresh_token` で更新
- サーバー側のみ `SUPABASE_SERVICE_ROLE_KEY` を利用し、クライアントには公開しない

#### Phase 3 — GSC API呼び出し
**Step 4** : サイト一覧取得APIの実装（`/api/sites`）
**Step 5** : GSC Analytics取得APIの実装（`/api/chat`にGSC呼び出しロジックを組み込む）

#### Phase 4 — チャットUI
**Step 6** : LLM（Claude API）とGSC APIを繋ぐチャットAPIの実装（`/api/chat`）
**Step 7** : フロントのチャットUI実装（サイト選択 + チャット画面）

#### Phase 5 — デプロイ
**Step 8** : Vercel環境変数設定・デプロイ・動作確認
